\chapter{RESULTS AND DISCUSSIONS}
\pagebreak


\begin{center}
{\LARGE\textbf{RESULTS AND DISCUSSIONS}}
\end{center}

This chapter presents the results and discussion of the implementation and testing of the TrapPot system. The main objective of this chapter is to evaluate the extent to which the developed system met its initial objective of developing an AI-enabled honeypot environment that is capable of harvesting SSH and Telnet communications, tracking the network traffic, classifying traffic behavior, and visualizing the collected data via a visualization interface. 

The analysis of the implemented system is performed as a complete pipeline and not as separate tools. TrapPot integrates Cowrie for honeypot interaction, Zeek for network monitoring, a Random Forest-based AI detector for traffic classification, and the ELK Stack for log storage and visualization. Therefore, the results are discussed according to the performance of these components in the course of testing.

The chapter is structured in four main parts. The first part describes the analysis strategy applied to the system assessment. The second section presents the results of the tests of the main components. The third part elaborates on the main findings from the results. The last section discusses the meaning of the results, the limitations of the current implementation and possible improvements for future work.

\section{Analysis Strategy} % (fold)
\label{sec:analysis_strategy}

In order to evaluate TrapPot from an operational perspective, the analysis strategy was set. Since the project is a cybersecurity monitoring and detection system, the evaluation is not only based on the accuracy of the machine learning. It also depends on whether the whole system pipeline works properly from the first honeypot interaction until the last detection output and the dashboard visualization. 

The analysis was developed based on the following main questions: 

\begin{itemize}
    \item Can Cowrie act as an SSH/Telnet honeypot and log the interactions of an attacker? 
    \item Can Zeek monitor the traffic generated by the honeypot and produce structured network logs?
    \item Can the AI detector read the \texttt{conn.log} file of Zeek and classify the traffic behavior with the trained Random Forest model? 
    \item Can the generated logs be processed with Logstash and sent to Elasticsearch?
    \item Can Kibana display the indexed data through data views and the TrapPot dashboard?
\end{itemize}

To answer these questions, the system was tested in a controlled local lab environment with Docker Compose. To check if it was deployed correctly, I first started all the services I needed and then used the docker commands to check the status. Then an SSH session was opened to the Cowrie honeypot on port 2222. During the test session, we signed in with the configured test credentials and ran some commands such as \texttt{whoami}, \texttt{pwd}, \texttt{ls -la}, \texttt{cat /etc/passwd}, and \texttt{wget}.

These commands were chosen because they represent common actions that an attacker may take after gaining access to a system. For example, \texttt{whoami} checks the current user, \texttt{pwd} checks the working directory, \texttt{ls -la} lists files, \texttt{cat /etc/passwd} attempts to read user-related system information, and \texttt{wget} simulates a file download attempt.

The output of each component was then checked. Cowrie logs were reviewed to confirm that the session and commands were recorded. Zeek logs were checked to confirm that were generated. The AI detector output was reviewed to verify that live classification alerts were generated. The Elasticsearch index was checked to confirm the logs were indexed and the Kibana was opened to confirm the data view and dashboard were available.

\section{Analysis Result} % (fold)
\label{sec:analysis_result}

The test results showed that the main parts of TrapPot worked well as a connected Docker-based system. Docker Compose could start the required services such as Cowrie, Zeek, the AI detector, Elasticsearch, Logstash, Kibana and the ELK setup service. This confirmed that the project can be deployed as a portable multi container environment.

Cowrie successfully accepted an SSH connection on port 2222. After login, the honeypot provided a fake Linux shell environment and accepted the test commands that had been entered during the session. Cowrie logs recorded login attempt, successful authentication, commands executed, and file download attempt. This result shows that the honeypot layer achieved its goal, that is to attract and capture attacker-like behavior without exposing a real production system.

Zeek was able to successfully monitor the traffic of Cowrie sessions. The connection records in the \texttt{conn.log} file showed SSH traffic from the host machine to the Cowrie container. Zeek also managed to capture traffic generated by the simulated \texttt{wget} command. This suggests that the network monitoring layer was working correctly and Zeek was able to turn traffic activity into structured log records.

The AI detector successfully read Zeek’s \texttt{conn.log} file and produced live classification alerts. During testing the detector showed classification output such as \texttt{Normal detected}. This further validates that the AI detector was able to use the Zeek logs to extract the relevant network traffic features, load the trained Random Forest model, and provide a classification result at runtime. The detector also writes detection records to \texttt{detections.json} so that the detection output can be stored for later analysis.

We validated Logstash and Elasticsearch too. Logstash was attached to Elasticsearch and parsed the existing TrapPot logs. Elasticsearch had indices related to TrapPot like \texttt{trappot-zeek-conn} and \texttt{trappot-detector}. This confirms that the log processing and indexing layer was working properly. 

Kibana was hosted on \texttt{localhost:5601}. Data views were available for Cowrie logs, Zeek connection logs, Zeek SSH logs, and AI detector results. The \texttt{TrapPot Overview} was also available in Kibana dashboard. This validated that the visualization layer was correctly configured and can be used to inspect the collected logs via a browser interface.

\section{Major Findings} % (fold)
\label{sec:major_findings}

The first major finding is that TrapPot managed to integrate honeypot interaction, network monitoring, machine learning detection and dashboard visualization into one working environment. The system was not only collecting terminal logs. It also transformed network activity into structured logs, and used the logs for live classification.

The second finding is that Zeek is crucial in linking the honeypot to the AI detector. Cowrie logs the attacker behavior on the interaction level, like login attempts and commands. Zeek records network activity at the level of traffic, including ports, protocols, services, duration, byte counts, packet counts, and connection state. This makes the system more useful because the AI detector can classify traffic based on structured network features, not just raw command logs.

Third, the AI detector was successfully incorporated into the live system. The detector loaded \texttt{trappot\_model.pkl} and \texttt{label\_encoder.pkl}, monitored Zeek’s \texttt{conn.log}, extracted the twelve required features, and produced live classification alerts. This confirms that the machine learning component was coupled with the implemented pipeline and not just an offline standalone model.

The fourth finding is that Docker Compose improved deployment and portability.  With a single Docker Compose command, you can start the full system instead of installing and configuring each component manually. This makes TrapPot easier to reproduce, test and distribute to other machines.

Fifth, the system became more usable with the Kibana. Terminal logs are handy for debugging, but Kibana provides a cleaner interface to search, filter and view activity via dashboards. This facilitates the analysis and presentation of the collected data.

\section{Discussion} % (fold)
\label{sec:discussion}

Results show that TrapPot attained its main implementation goals. The system recorded SSH honeypot activity, created network logs with Zeek, created AI classification alerts, stored logs in Elasticsearch and visualized the data with Kibana. This shows that the proposed architecture can function as a honeypot-based monitoring and detection system.

One important aspect is that the system makes use of two different sorts of observation. Cowrie logs the attacker's interaction with the fake system, and Zeek logs the related network traffic. The combination adds value to the collected data, providing both a behavioral and a network level perspective of the same activity. Then, the AI detector uses Zeek network features to classify traffic behavior.

Nevertheless there are some limitations to consider. For one, the testing was done in a controlled local environment, not on a public network. This was done for safety and ethical reasons, but it means that the system was not exposed to large amounts of real attacker traffic. Future testing may be conducted in an approved segregated lab network to capture more realistic traffic.

Second, the live AI detector provided a classification output, but there was limited observed testing activity. The SSH session tested is manually generated and small scale which might not fully represent the attack patterns in the IoT-23 training dataset. Additional varying attack simulations would be necessary to evaluate the model's robustness under different live traffic conditions.

Third, the Random Forest model shows high accuracy on the prepared IoT-23 dataset, however the dataset accuracy is not always a guarantee to the same performance in real deployment. In real environments, the network traffic may have patterns different from the training data. Further validation using real honeypot traffic and additional attack scenarios should be performed in future work.

Fourth, Elasticsearch, Logstash and Kibana consume more system resources than the core honeypot and AI detector components. They offer useful capabilities for visualization and storage. If the system is to be deployed for long term monitoring or higher traffic volume they may require tuning of memory and performance.

Overall, the results show that TrapPot is a successful proof of concept system. It demonstrates how Cowrie, Zeek, Random Forest classification and ELK Stack can be integrated in a Docker-based environment for monitoring and analysis of IoT honeypots. The system provides a good foundation for future improvements such as more sophisticated attack classification, bigger scale deployment, better model validation and more granular dashboard analytics.
