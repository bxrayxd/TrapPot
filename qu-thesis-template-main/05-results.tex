Did the system work? Yes, and no, and then yes again.

The final pipeline held together. But getting there required fixing a class-imbalance problem that took the first three training runs from a theoretical model to a real one, and verifying five separate components by hand before we trusted any of them. This chapter walks through what we checked, what we found, and where the gaps still are.

\section{Analysis Strategy}
\label{sec:analysis_strategy}

We evaluated TrapPot at two levels: the machine learning model and the operational pipeline.

Model accuracy alone does not tell you whether a honeypot system works. A classifier that scores 99.59\% on a dataset but cannot read a live Zeek log is useless. So we set two parallel success conditions: the Random Forest model had to hit at least 99\% accuracy on the IoT-23 hold-out set \cite{iot23dataset,karim2024efficient}, and every component in the stack had to pass a manual verification check.

The verification questions we used:

\begin{itemize}
\item Does Cowrie accept an SSH connection on port 2222 and log the session?
\item Does Zeek generate \texttt{conn.log} and \texttt{ssh.log} from that traffic?
\item Does the AI detector read \texttt{conn.log}, extract the twelve features, and produce a live classification?
\item Does Logstash process those logs and connect to Elasticsearch?
\item Does Kibana display the TrapPot data views and the Overview dashboard?
\end{itemize}

We ran the test by starting the full stack with \texttt{docker compose up --build -d}, then opening an SSH session to \texttt{localhost:2222} and running five commands: \texttt{whoami}, \texttt{pwd}, \texttt{ls -la}, \texttt{cat /etc/passwd}, and \texttt{wget}. Those five were chosen because they represent what a real attacker does in the first sixty seconds after gaining shell access, check who you are, check where you are, look for interesting files, and try to download something.

\section{Analysis Result}
\label{sec:analysis_result}

\subsection*{Cowrie}

It worked on the first try. Cowrie accepted the SSH connection on port 2222, gave us a fake Linux shell, and logged everything, the login attempt, each command, and the \texttt{wget} download attempt, in structured JSON. The fake filesystem looked convincing enough that the session felt like a real Linux device responding \cite{9071014,9359711}.

\subsection*{Zeek}

Zeek was the component we were least sure about (because the \texttt{network\_mode: "service:cowrie"} setup is not something you see in many tutorials). After the SSH session, \texttt{conn.log} and \texttt{ssh.log} appeared with the expected fields. The connection record showed SSH traffic between the host and the Cowrie container, and the \texttt{wget} traffic appeared as a separate entry. Twelve features, correctly structured. That was the result we needed.

\subsection*{AI Detector}

The detector read \texttt{conn.log}, extracted the features, and printed \texttt{Normal detected}, which is correct for a single manual SSH session from localhost. It also wrote the result to \texttt{detections.json}. The classification was not exciting, but it confirmed that the live path worked: model loaded, features extracted, prediction produced.

\subsection*{ELK Stack}

Logstash connected to Elasticsearch and processed the TrapPot logs. Elasticsearch contained the expected indices for Cowrie, Zeek, and detector output. Kibana was available at \texttt{localhost:5601} with all four data views and the TrapPot Overview dashboard \cite{9001068}.

\subsection*{Random Forest Model}

We trained on IoT-23 \cite{iot23dataset} with an 80/20 split and five-fold cross-validation. The first three runs came in around 72\%, not good enough, and the cause was class imbalance: brute-force records made up the majority of the training set, so the model was essentially ignoring malware download and command execution sessions. After applying SMOTE oversampling to balance the classes, accuracy on the hold-out set jumped to 99.59\%. Table~\ref{tab:model_accuracy} shows the final result.

\section{Major Findings}
\label{sec:major_findings}

Four things stand out.

\textbf{The pipeline held end-to-end.} Cowrie captured the session. Zeek produced the connection records. The AI detector classified them live. ELK stored and displayed everything. No component failed silently.

\textbf{Zeek is the critical link.} Cowrie sees terminal behavior. The classifier needs numbers. Zeek is what translates one into the other, and if \texttt{conn.log} does not exist or does not have the right fields, the detector has nothing to work with. Getting Zeek's network namespace setup right was probably the most technically difficult part of this project.

\textbf{Class imbalance changed the result.} A 72\% model that performs mainly on the majority class is not acceptable for a security application because it can miss malware download sessions. SMOTE fixed that failure mode, but the lesson is that you have to look at per-class metrics, not just accuracy alone \cite{saied2023comparative,karim2024efficient}.

\textbf{Docker Compose changed the deployment story.} One command starts the whole stack. That is not a small thing for a project that requires seven services with specific startup dependencies and shared volumes.

\section{Discussion}
\label{sec:discussion}

TrapPot achieved what it set out to do. The question is what that actually means.

The 99.59\% accuracy figure is real, but it is a dataset number. IoT-23 is a good dataset, labeled, Zeek-compatible, widely used in the literature \cite{iot23dataset,dekeersmaeker2023survey}, but it is not the same as traffic from real attackers hitting a real honeypot on a real network. We cannot claim the classifier would perform the same way on live data, and we are not claiming that.

What we can say is that the architecture is sound. The Cowrie → Zeek → Random Forest → ELK chain works. The model handles the five classes well on the prepared data. And the system is portable enough that the next version, the one with real network exposure and 30 days of attacker traffic, is buildable from what we have.

The main limitation is not the model. It is the test environment. One SSH session, five commands, localhost traffic. That proves the pipeline works, not that the system can handle real attacker traffic. That validation is the work that comes next \cite{famera2025mirai,rodriguezgomez2025systematic}.
