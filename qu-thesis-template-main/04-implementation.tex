\section{Development Environment}
\label{sec:dev_environment}
% ============================================================

% ------------------------------------------------------------
\subsection{Primary Platform}
\label{subsec:primary_platform}
% ------------------------------------------------------------

Python 3.10+ handles everything custom: feature extraction, model training, and live classification \cite{optimized2024arxiv,karim2024efficient}. We chose Python because scikit-learn, pandas, NumPy, and joblib are the standard toolkit for this kind of work, and all four are well-documented enough that debugging is manageable \cite{dimauro2021supervised,lee2021classification}.

scikit-learn 1.3+ provides the Random Forest implementation, train-test splitting, cross-validation, and metrics \cite{lee2021classification,saied2023comparative,karim2024efficient}. pandas and NumPy handle the Zeek log data, parsing tab-separated \texttt{conn.log} rows, filling missing values, and arranging features in the order the trained model expects.

% ------------------------------------------------------------
\subsection{Containerization}
\label{subsec:containerization}
% ------------------------------------------------------------

Docker and Docker Compose run the full TrapPot stack \cite{9359711,commey2025blockchain}. The Compose file starts eight services: Cowrie, Zeek, the AI detector, Elasticsearch, Logstash, Kibana, \texttt{elastic\_setup}, and \texttt{kibana\_setup}. Startup order is managed through health checks and service dependencies, Elasticsearch has to be healthy before the setup containers run, and Kibana has to be ready before the dashboard import script executes.

Why Docker? Because installing Cowrie, Zeek, Elasticsearch 9.4.0, Logstash, and Kibana manually on a single machine, with matching versions and shared volumes, is a multi-day project. With Docker Compose, it is one command: \texttt{docker compose up --build -d}.

% ------------------------------------------------------------
\subsection{Key Tools}
\label{subsec:data_tools}
% ------------------------------------------------------------

\textbf{Cowrie} (SSH/Telnet honeypot layer) \cite{9071014,9359711,9558948}: logs authentication attempts, commands, sessions, and download attempts in structured JSON. Official Docker image: \texttt{cowrie/cowrie}.

\textbf{Zeek} (network monitoring layer): produces \texttt{conn.log} and \texttt{ssh.log} with the twelve connection-level features the detector reads. Docker image: \texttt{blacktop/zeek}, run with \texttt{-C} to disable checksum validation (required in containerized environments).

\textbf{Elastic Stack 9.4.0} (storage and visualization) \cite{9001068,log2evt2025}: Elasticsearch stores TrapPot indices, Logstash parses and forwards logs from all three sources, Kibana provides data views and the TrapPot Overview dashboard at \texttt{localhost:5601}.

\section{System Architecture}
\label{sec:system_architecture}
% ============================================================

Four functions. One pipeline. Capture → Monitor → Classify → Visualize \cite{franco2021survey,rodriguezgomez2025systematic,lanz2025optimizing}.

\subsection{Architecture Overview}
\label{subsec:high_level_arch}

\textbf{Capture layer:} Cowrie on TCP 2222 (SSH) and TCP 2223 (Telnet). Records every login attempt, command, session event, and file transfer in JSON \cite{9071014,log2evt2025}.

\textbf{Network monitoring layer:} Zeek in Cowrie's network namespace. Converts traffic to structured rows in \texttt{conn.log} and \texttt{ssh.log}. This is the input the AI detector actually reads.

\textbf{Intelligence layer:} The AI detector watches \texttt{conn.log}, extracts twelve features per row, loads \texttt{trappot\_model.pkl} and \texttt{label\_encoder.pkl}, and writes predictions to \texttt{detections.json}.

\textbf{Storage and visualization layer:} Logstash reads all three log sources and forwards them to Elasticsearch. Kibana reads Elasticsearch and shows the TrapPot Overview dashboard.

% ------------------------------------------------------------
\subsection{Data Flow}
\label{subsec:component_interaction}
% ------------------------------------------------------------

Real-time. When a connection hits the honeypot, Cowrie logs the interaction and Zeek logs the network event, concurrently, not sequentially. The AI detector runs in a loop: read new \texttt{conn.log} rows, extract features, predict, write to \texttt{detections.json}. Logstash ships the logs to Elasticsearch on a polling schedule. Kibana refreshes the dashboard from Elasticsearch.

% ------------------------------------------------------------
\subsection{Network Topology}
\label{subsec:network_topology}
% ------------------------------------------------------------

Everything inside Docker's bridge network. Zeek uses \texttt{network\_mode: "service:cowrie"} to share Cowrie's network namespace, that is how it sees the honeypot traffic without running as a separate network sniffer. External exposure: 2222, 2223, 5601. Nothing else.

TrapPot is only safe in an approved, isolated lab environment. Not connected to university production infrastructure. Not exposed to the public internet.

\section{Component Implementation}
\label{sec:component_implementation}
% ============================================================

% ------------------------------------------------------------
\subsection{Cowrie Deployment}
\label{subsec:cowrie_deployment}
% ------------------------------------------------------------

Cowrie runs from the official \texttt{cowrie/cowrie} Docker image \cite{9071014,9359711}. Two configuration files are mounted into the container: \texttt{cowrie.cfg} (main configuration) and \texttt{userdb.txt} (login rules).

What the configuration does:

\begin{itemize}
\item Exposes SSH on host port 2222 and Telnet on 2223.
\item Sets a realistic Linux identity, custom hostname, OS release string, \texttt{/etc/issue}, MOTD, and a populated fake filesystem.
\item Outputs structured JSON logs for authentication attempts, commands, and file activity.
\item Persists logs to a shared Docker volume that Logstash reads.
\end{itemize}

% ------------------------------------------------------------
\subsection{Zeek Deployment}
\label{subsec:zeek_deployment}
% ------------------------------------------------------------

Zeek uses the \texttt{blacktop/zeek} image, attached to Cowrie's network namespace. The \texttt{-C} flag disables checksum validation, without it, Zeek drops most packets in Docker because the container networking stack offloads checksum computation. We spent more time on this single flag than on any other configuration issue.

Zeek also receives the local Docker address range on startup so it correctly classifies lab network traffic as ``local'' rather than tagging it as an external source.

% ------------------------------------------------------------
\subsection{AI Detector}
\label{subsec:ai_detector}
% ------------------------------------------------------------

The AI detector is a Python script that runs in its own Docker container. On startup it loads \texttt{trappot\_model.pkl} and \texttt{label\_encoder.pkl}, then enters a monitoring loop over Zeek's \texttt{conn.log}.

\begin{algorithm}[H]
\caption{Real-Time Classification Loop}
\label{alg:realtime_classification}
\begin{algorithmic}[1]
\REQUIRE Trained model $M$, encoder $LE$, Zeek log \texttt{conn.log}
\ENSURE Live classification alerts and detection records in \texttt{detections.json}
\STATE $M \leftarrow$ load\_model(\texttt{trappot\_model.pkl})
\STATE $LE \leftarrow$ load\_encoder(\texttt{label\_encoder.pkl})
\STATE Wait until \texttt{conn.log} exists
\STATE Open \texttt{conn.log} in tail mode
\WHILE{True}
\STATE Read next new line
\IF{no new line} \STATE Wait briefly; continue \ENDIF
\IF{line starts with ``\#''} \STATE Skip header; continue \ENDIF
\STATE $F \leftarrow$ extract\_features(line) \COMMENT{12 features}
\STATE Preprocess numerical and categorical values
\STATE $prediction \leftarrow M$.predict($F$)
\STATE Print live alert with predicted class
\STATE Append result to \texttt{detections.json}
\ENDWHILE
\end{algorithmic}
\end{algorithm}

The detector reads only new lines, it does not reprocess the whole file on each iteration. That keeps CPU usage low during quiet periods and ensures the classification latency matches Zeek's log-write frequency.

% ------------------------------------------------------------
\subsection{ELK Stack Configuration}
\label{subsec:elk_configuration}
% ------------------------------------------------------------

Elastic Stack 9.4.0. Three Logstash pipelines, one per log source (Cowrie JSON, Zeek TSV, detector JSON), each with a distinct input path, field mappings, and output index name \cite{9001068,log2evt2025}.

The \texttt{elastic\_setup} container runs after Elasticsearch is healthy and creates the TrapPot user, role, index templates, and ILM policy. The \texttt{kibana\_setup} container runs after Kibana is ready and imports the data views and dashboard objects via the Kibana API. This means the dashboard is available automatically on first deployment, no manual setup needed.

\section{Model Training Pipeline}
\label{sec:model_training_pipeline}
% ============================================================

We trained the Random Forest model offline on IoT-23 before deployment \cite{iot23dataset}. The training script runs outside Docker, it is a standalone Python script that reads the dataset, preprocesses it, trains the model, and saves \texttt{trappot\_model.pkl} and \texttt{label\_encoder.pkl}.

\textbf{Data preparation:}

\begin{itemize}
\item Load IoT-23 Zeek-compatible traffic flows \cite{iot23dataset,dekeersmaeker2023survey}.
\item Select the twelve features: \texttt{id.orig\_p}, \texttt{id.resp\_p}, \texttt{proto}, \texttt{service}, \texttt{duration}, \texttt{orig\_bytes}, \texttt{resp\_bytes}, \texttt{orig\_pkts}, \texttt{resp\_pkts}, \texttt{orig\_ip\_bytes}, \texttt{resp\_ip\_bytes}, \texttt{conn\_state}.
\item Encode categorical features (\texttt{proto}, \texttt{service}, \texttt{conn\_state}) with \texttt{LabelEncoder}.
\item Fill missing values with column medians (duration, byte counts) or mode (service).
\item Apply SMOTE to balance class frequencies across the five attack classes \cite{9213728,9912910,abdulsalam2023multiclass}.
\item 80/20 train-test split; 5-fold cross-validation on the training set \cite{optimized2024arxiv,karim2024efficient}.
\end{itemize}

\textbf{Training result:} Random Forest achieved 99.59\% accuracy on the hold-out set after SMOTE. The first three runs, before rebalancing, came in around 72\%. That gap is the SMOTE contribution.

\textbf{Saved artifacts:}
\begin{itemize}
\item \texttt{trappot\_model.pkl}, the trained \texttt{RandomForestClassifier} object (scikit-learn 1.3+).
\item \texttt{label\_encoder.pkl}, the \texttt{LabelEncoder} for decoding numeric class predictions back to names.
\end{itemize}

\section{Deployment and Verification}
\label{sec:integration_deployment}
% ============================================================

Start the whole stack:

\begin{center}
\texttt{docker compose up --build -d}
\end{center}

After startup, connect to Cowrie on port 2222 (SSH) or 2223 (Telnet), and open Kibana at \texttt{http://localhost:5601}. Credentials are in \texttt{TrapPot/.env}.

% ------------------------------------------------------------
\subsection{Deployment Checks}
\label{subsec:deployment_verification}
% ------------------------------------------------------------

\begin{itemize}
\item \texttt{docker compose ps}, confirm all eight containers are running.
\item SSH to \texttt{localhost:2222}, confirm Cowrie accepts the connection and logs it.
\item Check \texttt{conn.log} and \texttt{ssh.log}, confirm Zeek wrote records after the SSH session.
\item Check AI detector container logs, confirm live classification output appears.
\item Check Elasticsearch, confirm TrapPot indices exist.
\item Open Kibana at \texttt{localhost:5601}, confirm data views and TrapPot Overview dashboard load.
\end{itemize}

\section{Testing Strategy}
\label{sec:testing_strategy}
% ============================================================

We tested four ways: unit, integration, performance, and classifier accuracy \cite{confusion2023pmc,mary2024network,hindy2023comprehensive}.

% ------------------------------------------------------------
\subsection{Unit Tests}
\label{subsec:unit_testing}
% ------------------------------------------------------------

\textbf{Cowrie:} SSH to port 2222, run \texttt{whoami}, \texttt{pwd}, \texttt{ls -la}. Confirmed shell emulation and JSON log output.

\textbf{Zeek:} Checked that \texttt{conn.log} and \texttt{ssh.log} appeared after the SSH session. Both did.

\textbf{AI detector:} Reviewed container logs after Zeek produced connection entries. The detector read \texttt{conn.log}, extracted features, and printed classification results.

\textbf{ELK:} Checked that Elasticsearch indices existed and Kibana loaded the dashboard.

% ------------------------------------------------------------
\subsection{Integration Tests}
\label{subsec:integration_testing}
% ------------------------------------------------------------

\textbf{Cowrie → Zeek:} SSH to port 2222. \texttt{conn.log} contained the resulting SSH connection record.

\textbf{Zeek → AI detector:} New \texttt{conn.log} entries triggered live detector output within seconds.

\textbf{Logstash → Elasticsearch:} Logstash processed all three log sources and forwarded them to the correct indices.

\textbf{Elasticsearch → Kibana:} Kibana read the indices and showed data in the TrapPot data views and dashboard.

% ------------------------------------------------------------
\subsection{Performance Test}
\label{subsec:performance_testing}
% ------------------------------------------------------------

The stack ran on a single development machine. Zeek produced connection logs after each SSH session. The detector classified new records without noticeable delay. Kibana stayed accessible throughout. The ELK containers are the memory constraint, the host machine needs to provide enough RAM for Elasticsearch and Kibana to start cleanly. Cowrie and the AI detector are negligible by comparison.

% ------------------------------------------------------------
\subsection{Security Test}
\label{subsec:security_testing}
% ------------------------------------------------------------

No public exposure. The Docker Compose deployment exposes ports 2222, 2223, and 5601 only, as shown in the project \texttt{docker-compose.yml}. All attacker behavior was simulated locally using manual SSH sessions. The documentation makes clear that TrapPot must only run in a lab network with appropriate isolation from university production systems.

% ------------------------------------------------------------
\subsection{Classifier Accuracy Validation}
\label{subsec:classifier_validation}
% ------------------------------------------------------------

The trained Random Forest model was evaluated on the 20\% IoT-23 hold-out set \cite{iot23dataset}. The training process used the twelve Zeek-compatible features, SMOTE rebalancing, and scikit-learn's cross-validated Random Forest. Final accuracy: 99.59\%. The model was saved and deployed in the AI detector container. Future validation with live honeypot traffic and diverse attack scenarios remains as described in Chapter~6.
