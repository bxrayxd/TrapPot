{\LARGE\textbf{INTRODUCTION}}
\end{center}

% ─────────────────────────────────────────────────────────────────────────────

In 2016, Mirai compromised more than 600,000 IoT devices by trying 62 default credential pairs against exposed Telnet and SSH services \cite{203628}. Most of those devices were cameras and routers that had not been patched, on networks where nobody expected them to become weapons. The result was one of the largest DDoS attacks ever recorded. Saudi Arabia is currently deploying IoT devices at scale across smart-city projects and digital infrastructure under Vision 2030, and many IoT environments still face similar exposure: weak credentials, limited patching, and remote access services left open by default.

TrapPot is our attempt to study that problem from inside the implemented Cowrie, Zeek, Random Forest, and ELK pipeline described in Chapter~\ref{sec:system_architecture}.

The idea is simple: instead of hardening real devices, we put a fake one on the network and watch who attacks it. Cowrie emulates an SSH and Telnet server that looks like a vulnerable embedded Linux device. Attackers connect, try to log in, and run commands while the system records the interaction. Zeek sits behind Cowrie and converts the network traffic into structured connection records. A Random Forest classifier reads those records and labels each connection: brute-force, scanning, malware download, command execution, or normal. The ELK Stack (Elasticsearch, Logstash, and Kibana) stores the logs and shows them in a dashboard at \texttt{localhost:5601}.

That is the system: build it, run it, and verify that the pipeline holds.

\section{Challenges}
\label{sec:challenges}
% ============================================================

The hardest part was not any single tool. It was making five tools talk to each other correctly inside Docker without any of them becoming a security risk.

We expected more practical tutorials for IoT honeypots than we found. Most of the guidance was either too high-level (``deploy a honeypot'') or too specific to production environments we did not have access to \cite{franco2021survey,rodriguezgomez2025systematic}. We ended up cross-referencing academic papers with official documentation and fixing problems through direct testing.

Isolation was the other constraint. A honeypot that is not properly contained can become an attack platform, not just a research tool \cite{commey2025blockchain}. We kept ports 2222, 2223, and 5601 as the only exposed surfaces, ran everything behind Docker's bridge network, and documented that TrapPot should only ever be used in an approved lab network. The whole design assumed that safety comes before realism.

The third challenge was keeping the data chain intact. Cowrie interaction, Zeek traffic records, AI detector output, and ELK indices had to stay connected. If Zeek's \texttt{conn.log} format changed or Logstash failed to parse a field, the classifier would see invalid input. Getting that chain to stay clean across all five components took most of the integration work.

\section{Study Scope}
\label{sec:study_scope}
% ============================================================

TrapPot focuses on SSH and Telnet. Not HTTP, not MQTT, not CoAP. Just the two protocols that Mirai-style botnets target most aggressively \cite{famera2025mirai,zhang2023identification}.

The classifier handles five classes: brute-force, scanning, malware download, command execution, and normal traffic. We trained it on IoT-23 \cite{iot23dataset} and targeted at least 99\% accuracy, a threshold supported by comparable work in the literature \cite{lee2021classification,karim2024efficient}. The system runs as a single-node Docker Compose deployment on a local lab network, isolated from production infrastructure. Deep learning methods, distributed deployment, and production-grade hardening are outside this scope and reserved for future work.

Why Random Forest? Lee et al.\ used it on Cowrie honeypot data in IoT environments and hit 96\% accuracy with 0.1~ms inference latency \cite{lee2021classification}. That result, from data that looks a lot like ours, was a stronger reason to choose it than any theoretical argument.

\section{Goals and Objectives}
\label{sec:goals_and_objectives}
% ============================================================

The project aims to improve IoT security readiness by observing attacker behavior, analyzing network traffic, and classifying suspicious activity. The main objectives are:

\begin{itemize}
\item \textbf{Understanding IoT Attack Behavior:} Watch how attackers interact with emulated SSH and Telnet services, including what credentials they try, what commands they run, and how long they stay.
\item \textbf{Improving IoT Attack Classification:} Build a Random Forest classifier that labels traffic into five classes with at least 99\% accuracy on the prepared dataset.
\item \textbf{Supporting Proactive Security Measures:} Give security teams earlier visibility into attack behavior so they can act before damage occurs.
\item \textbf{Enhancing IoT Threat Awareness:} Present captured activity through Kibana dashboards in a form that researchers and administrators can actually read and act on.
\item \textbf{Reducing the Attack Surface of IoT Environments:} Use observed attack patterns to inform practical recommendations, including better access controls and fewer exposed services.
\end{itemize}
