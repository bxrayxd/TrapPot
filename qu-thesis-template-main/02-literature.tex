This chapter reviews the research that shaped TrapPot's design. The review is organized around three questions: what threats do IoT environments face, what approaches already exist for honeypot-based detection, and what tools and concepts underpin the system we built. The answers to those questions explain why TrapPot uses Cowrie, Zeek, Random Forest, and the ELK Stack rather than any of the other combinations the literature suggests \cite{franco2021survey,rodriguezgomez2025systematic,lanz2025optimizing}.

\subsubsection{IoT Security Vulnerabilities and the Threat Landscape}
\label{subsub:iot_vulnerabilities}
% ------------------------------------------------------------

The numbers are blunt. Mahmood and colleagues found that 55\% of IoT devices lacked encryption protection, with authentication failures appearing at the device, network, and application layers simultaneously \cite{mahmood2024exploring}. Islam and colleagues examined more than 11,000 IoT code samples and found that 39.58\% contained real Common Vulnerabilities and Exposures entries, with nearly 29\% of those being denial-of-service errors \cite{islam2023large}.

For TrapPot's context, the useful data point is behavioral. Mirai compromised more than 600,000 devices by trying 62 default credential pairs against exposed Telnet and SSH services \cite{203628}. Its successors, including Gafgyt and Mozi, use related credential-driven methods. The attack is not complex. It is fast, automated, and reliable because the underlying condition (default credentials and weak patching) continues to appear in IoT environments \cite{suareztangil2023stories}. That is what TrapPot is designed to observe.

% ------------------------------------------------------------
\subsubsection{Honeypot Deployment Challenges}
\label{subsub:honeypot_challenges}
% ------------------------------------------------------------

Deploying a honeypot is not as simple as running a container and waiting. Three problems. Each one shaped a decision.

First: fingerprinting. Attackers can detect honeypots by analyzing response timing, banner details, and behavioral quirks that do not match real devices \cite{franco2021survey}. A Cowrie instance that responds too consistently or presents an implausible filesystem is detectable. We cannot fully solve this problem in a lab-environment project, but we can reduce the most obvious signals through realistic configuration \cite{morozov2024sweet}.

Second: isolation. A honeypot that is not properly contained can become an attack relay \cite{commey2025blockchain}. If an attacker gains unexpected capabilities, a poorly isolated deployment can be used to target external systems. The Docker bridge network approach in TrapPot is a direct response to this: only three ports are exposed (2222, 2223, 5601), outbound traffic from the honeypot containers is blocked at the host firewall, and Cowrie's medium-interaction design ensures that commands run inside the emulated shell, not on the host OS.

Third: scale. Fixed honeypot designs struggle in IoT environments where device diversity is high \cite{morozov2024sweet}. For this project, we accepted that constraint and chose a medium-interaction SSH/Telnet honeypot, which covers the most attacked protocols without trying to emulate every IoT device type.

% ------------------------------------------------------------
\subsubsection{Machine Learning Classification Challenges}
\label{subsub:ml_challenges}
% ------------------------------------------------------------

Two problems dominate the literature on ML-based IoT intrusion detection: class imbalance and feature quality \cite{hindy2023comprehensive,dekeersmaeker2023survey}.

Class imbalance is the bigger problem in practice \cite{hindy2023comprehensive}. In real IoT traffic, brute-force and scanning events are far more common than malware downloads or command execution sessions. A classifier trained on imbalanced data learns to predict the majority class well and ignore everything else. Hindy et al.\ identified this as a primary failure mode in deployed IDS systems. TrapPot hit this problem directly: the first three training runs on raw IoT-23 data came in around 72\% \cite{9213728,9912910,abdulsalam2023multiclass}.

Feature quality is the second problem \cite{dekeersmaeker2023survey}. Irrelevant features do not just reduce accuracy; they make the model harder to interpret and slower to run. We chose twelve features from Zeek's \texttt{conn.log} because they are directly interpretable, fast to extract, and cover the main discriminating signals for the five classes. Several IoT intrusion detection studies report Random Forest accuracy above 99\% with similar compact feature sets \cite{alzahrani2025lightweight}.

% ------------------------------------------------------------
\subsubsection{Data Collection and Preprocessing Challenges}
\label{subsub:data_challenges}
% ------------------------------------------------------------

Labeling is hard. Rodriguez-Gomez and colleagues noted that manual attack labeling is time-consuming, inconsistent, and error-prone when attack patterns are ambiguous \cite{rodriguezgomez2025systematic}. Public IoT datasets differ in structure, feature definitions, and label schemes, which makes it difficult to train on one and deploy on another \cite{dekeersmaeker2023survey}.

This is why we chose IoT-23 for training (a well-labeled, Zeek-compatible dataset) and live Zeek output for deployment \cite{iot23dataset,alex2023comprehensive}. The feature format has to match between training and live inference. If it does not, the model receives invalid input. We chose IoT-23 specifically because it was generated from Zeek logs and its feature columns align with what the live detector reads from \texttt{conn.log}.

\subsection{Related Work}
\label{sub:related_work1}
% ------------------------------------------------------------

Six papers. These are the ones that shaped TrapPot. Not coincidentally, they also validate the three main design choices: Cowrie as the honeypot layer, Random Forest as the classifier, and the ELK Stack for visualization.

\textbf{Lee et al.\ (2021)} used Cowrie in IoT smart factory environments and trained a Random Forest classifier on brute-force, scanning, and malware download classes. They hit 96\% accuracy with 0.1~ms inference latency \cite{lee2021classification}. This is the closest prior work to TrapPot's architecture. The main difference: TrapPot adds a fifth class (command execution), uses IoT-23 rather than their factory-specific dataset, and packages everything in Docker Compose for portability.

\textbf{Alajrami and Abu-Naser (2025)} built a Cowrie-based IDS that tested Random Forest alongside Logistic Regression and Isolation Forest for SSH/Telnet attacks \cite{alajrami2025aipowered}. They found Random Forest competitive with more complex approaches on similar data. TrapPot keeps the single-model design rather than comparing classifiers because explainability and simplicity matter for an undergraduate project.

\textbf{Saied et al.\ (2023)} reported 99.99\% Random Forest accuracy for multi-class IoT botnet classification, outperforming SVM, KNN, and decision trees on the same dataset \cite{saied2023comparative}. This result, combined with Lee et al., makes the algorithm choice straightforward.

\textbf{Alzahrani et al.\ (2025)} developed lightweight Random Forest methods with feature selection optimized for IoT resource constraints, achieving 98\% accuracy \cite{alzahrani2025lightweight}. Their approach supports the idea that a compact feature set, twelve features rather than fifty, does not necessarily sacrifice accuracy.

\textbf{Saputro et al.\ (2021)} measured Cowrie resource overhead in IoT honeypot deployments and found CPU usage below 6.3\% \cite{9359711}. That result matters for TrapPot because Cowrie has to share a host with Elasticsearch, Logstash, and Kibana, which are more resource-hungry.

\textbf{Lanz et al.\ (2025)} reviewed ML optimization for IoT honeypots and identified supervised learning with Random Forest as the most effective approach for honeypot attack classification, with accuracy reaching 96\% \cite{lanz2025optimizing}. Rodriguez-Gomez et al.\ made the complementary point: manual log analysis does not scale, which is why automated classification is necessary \cite{rodriguezgomez2025systematic}.

Together these papers answer the ``why'' questions that would otherwise be arbitrary choices: why Cowrie, why Random Forest, why twelve features, why IoT-23.

\subsection{Technical Background}
\label{sub:technical_background}
% ------------------------------------------------------------

This section defines the tools and concepts TrapPot is built on. Readers who know these tools well can skim; the goal is to be precise about what each component does and why.

% ------------------------------------------------------------
\subsubsection{Honeypots}
\label{subsub:honeypots}
% ------------------------------------------------------------

A honeypot is a system designed to look vulnerable and attract attackers while recording what they do \cite{9268794}. Three interaction levels are common: low (minimal emulation, mostly logging), medium (believable shell interaction, controlled environment), and high (close to a real system, higher risk). Medium-interaction honeypots are common in research because they provide enough realism to attract sustained attacker behavior without making containment too difficult \cite{9268794,franco2021survey}.

% ------------------------------------------------------------
\subsubsection{Cowrie}
\label{subsub:cowrie}
% ------------------------------------------------------------

Cowrie is an open-source medium-interaction SSH and Telnet honeypot \cite{9071014}. It emulates a UNIX-like shell, records login attempts, executed commands, downloaded files, and session metadata, and outputs structured JSON. That JSON structure is what Logstash reads and what the detection chain starts from. Cowrie does not execute commands on the host; the shell is simulated \cite{9071014,9359711}.

% ------------------------------------------------------------
\subsubsection{SSH and Telnet}
\label{subsub:ssh_telnet}
% ------------------------------------------------------------

SSH provides encrypted remote access. Telnet provides plaintext remote access. Both remain primary attack targets in IoT environments because automated bots, including Mirai, try default credentials against any open port 22 or port 23 they find \cite{9558948,203628}. This is why TrapPot listens on ports 2222 (SSH) and 2223 (Telnet) and nothing else.

% ------------------------------------------------------------
\subsubsection{ELK Stack}
\label{subsub:elk_stack}
% ------------------------------------------------------------

Elasticsearch, Logstash, Kibana \cite{9001068}. Logstash ingests and parses log data. Elasticsearch indexes and stores it. Kibana provides the web interface for querying and visualizing it. TrapPot uses Elastic Stack version 9.4.0. Logstash ships Cowrie JSON, Zeek logs, and AI detector output to separate Elasticsearch indices. Kibana displays all three in the TrapPot Overview dashboard at \texttt{localhost:5601}.

% ------------------------------------------------------------
\subsubsection{Random Forest}
\label{subsub:random_forest}
% ------------------------------------------------------------

An ensemble of decision trees. Each tree trains on a random data subset and a random feature subset. Final prediction: majority vote. Every tree gets a say \cite{lee2021classification,saied2023comparative}. Random Forest reduces overfitting compared to a single tree, handles structured high-dimensional data well, and produces fast predictions at inference time. Published IoT IDS studies report accuracy in the 96–99.99\% range with Random Forest on similar datasets.

% ------------------------------------------------------------
\subsubsection{Zeek}
\label{subsub:Zeek}
% ------------------------------------------------------------

Zeek is a passive network analysis framework \cite{9001068}. It does not match packet signatures; it extracts metadata from traffic and writes it to structured log files. The output TrapPot uses is \texttt{conn.log} (connection-level summary: source/destination ports, bytes, packets, duration, protocol, service, state) and \texttt{ssh.log} (SSH-specific metadata). These logs are what the AI detector reads.

Zeek runs in Cowrie's network namespace using \texttt{network\_mode: "service:cowrie"} in Docker Compose, with checksum validation disabled (\texttt{-C}) because containerized packet capture breaks checksum verification by default.

% ------------------------------------------------------------
\subsubsection{Supervised Learning and Key Metrics}
\label{subsub:supervised_learning}
% ------------------------------------------------------------

Supervised learning means the model learns from labeled examples and applies those patterns to new data \cite{hopf2021supervised,dimauro2021supervised}. TrapPot's Random Forest is supervised: it learns from labeled IoT-23 records and predicts classes for new Zeek connection records.

Metrics \cite{confusion2023pmc,mary2024network}:

\begin{itemize}
\item \textbf{Accuracy}: correct predictions / total predictions. Primary metric for TrapPot (target: 99\%+).
\item \textbf{Precision}: of all predicted positives for a class, how many are correct.
\item \textbf{Recall}: of all actual positives for a class, how many did the model find.
\item \textbf{F1-score}: harmonic mean of precision and recall. Useful when class frequencies differ.
\item \textbf{Confusion matrix}: shows exactly which class gets mislabeled as which other class.
\end{itemize}

\subsubsection{The Five Attack Classes}
\label{subsub:attack_types}
% ------------------------------------------------------------

TrapPot classifies five traffic types:

\begin{itemize}
\item \textbf{Brute Force}: repeated failed login attempts, typically automated credential stuffing \cite{203628,famera2025mirai}.
\item \textbf{Scanning}: port or service probing without authentication, identifying open services and versions \cite{zhang2023identification,suareztangil2023stories}.
\item \textbf{Malware Download}: file upload or download attempt after gaining access \cite{9071014,9558948}.
\item \textbf{Command Execution}: successful login followed by shell commands \cite{9071014,lee2021classification}.
\item \textbf{Normal}: baseline traffic, benign connections, polling, and protocol checks with no attack indicators \cite{lee2021classification}.
\end{itemize}
