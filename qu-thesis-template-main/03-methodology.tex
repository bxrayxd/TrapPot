\begin{center}
{\LARGE\textbf{METHODOLOGY}}
\end{center}

This chapter describes the systematic approach used to design, implement, and evaluate the TrapPot system \cite{franco2021survey,rodriguezgomez2025systematic,lanz2025optimizing}. The methodology encompasses the research design, data collection procedures, preprocessing and feature engineering pipeline, Random Forest classifier training and optimization, validation plan, and ethical considerations \cite{lee2021classification,hindy2023comprehensive,karim2024efficient}. The chapter is organized to guide the reader from the research design framework through each operational phase, culminating in the evaluation criteria that will assess whether TrapPot achieves its stated objective of capturing and automatically classifying IoT-targeted attacks with at least 99\% accuracy \cite{lee2021classification,karim2024efficient,kalaria2024iotpredictor}.

% ============================================================
\section{Research Design}
\label{sec:research_design}
% ============================================================

The TrapPot study was conducted using a Case Study research design supported by experimental elements through its controlled deployment and testing, and it follows a Quantitative approach \cite{franco2021survey,rodriguezgomez2025systematic}. This framework aligns with the project's goal of developing an AI-powered IoT honeypot system, providing a structured examination of how the system captures, processes, and classifies real attacker behavior \cite{lee2021classification,lanz2025optimizing}. This framework ensures that the system's performance both as a honeypot and as a machine-learning classifier is evaluated under realistic IoT attack conditions \cite{albaseer2024fedpot,fan2019honeydoc}. To ground this framework, the study is defined through its research design type and its analytical approach.

% ------------------------------------------------------------
\subsection{Research Design Type: Case Study}
\label{subsec:case_study}
% ------------------------------------------------------------

The primary research design is a Case Study, incorporating experimental elements through the controlled deployment and testing of the TrapPot system \cite{franco2021survey,morozov2024sweet}. The project adopts an in-depth Case Study design that focuses on a single instance: the TrapPot system deployed in a dedicated, isolated network environment \cite{albaseer2024fedpot,commey2025blockchain}. This design is suitable because the project aims to closely examine how a practical combination of medium-interaction honeypot technology and machine learning behaves when exposed to real-world IoT-targeted attacks \cite{lee2021classification,9359711}.

By concentrating on one concrete deployment, the Case Study design allows detailed observation of \cite{rodriguezgomez2025systematic,lanz2025optimizing}:

\begin{itemize}
    \item How attackers interact with the emulated Telnet and SSH services \cite{9071014,9558948,herwig2021analyzing},
    \item How logs flow through the ELK-based data pipeline \cite{9001068,log2evt2025}, and
    \item How the Random Forest classifier responds to different attack patterns (brute force, scanning, malware download,normal attack, and command execution) \cite{lee2021classification,karim2024efficient,saied2023comparative}.
\end{itemize}

This depth of analysis helps reveal the practical challenges, limitations, and successes of integrating honeypots with AI for threat intelligence, which is more informative for this type of system-development project than a broad survey or purely theoretical analysis \cite{franco2021survey,lanz2025optimizing}.

% ------------------------------------------------------------
\subsection{Research Approach: Quantitative}
\label{subsec:quantitative}
% ------------------------------------------------------------


The research uses Quantitative methods for validation purposes as described in \cite{hindy2023comprehensive,mary2024network}. The quantitative method works because all recorded data exists as numbers which follow a structured pattern which enables researchers to perform statistical assessments of model performance \cite{karim2024efficient,confusion2023pmc}. The research data consists of machine-generated structured information which includes honeypot logs that record authentication attempts and session lengths and command activities and file transfer data and network protocol details \cite{9071014,log2evt2025}. The system converts log data into numerical feature vectors which scientists use for statistical analysis according to \cite{dimauro2021supervised,ahsan2024effects}.

The methodology follows a system development and experimental validation pattern \cite{lee2021classification,rodriguezgomez2025systematic}:

\begin{itemize}
	\item Development and deployment of TrapPot (Cowrie honeypot + ELK Stack + Zeek + Random Forest classifier) in an isolated network \cite{9359711,commey2025blockchain}.
	\item The collection of real-time attack traffic data needs to occur during a specific time span \cite{lee2021classification,suareztangil2023stories}.
	\item Training and testing of the Random Forest model using the collected dataset \cite{saied2023comparative,karim2024efficient}.
	\item Evaluation process employs standard machine learning assessment methods which include accuracy measurements together with precision and recall values and F1-score calculations and confusion matrix evaluation \cite{confusion2023pmc,mary2024network}.
\end{itemize}

The quantitative research approach allows scientists to assess TrapPot classification accuracy and system performance during actual cyber attacks as described in \cite{hindy2023comprehensive,kalaria2024iotpredictor}.

As an AI-powered IoT honeypot system, TrapPot's core function is to lure attackers and collect actionable threat intelligence \cite{rodriguezgomez2025systematic,lanz2025optimizing}. Data collection in this project occurs in two distinct phases, each serving a specific purpose in the development lifecycle:

\begin{itemize}
    \item 3.2.1 – The data needed before and during system development, gathered from existing research and public datasets.
    \item 3.2.2 – The data that will be collected after deployment from the operational honeypot, used for real-time classification and validation.

% ------------------------------------------------------------
\subsection{Pre-Implementation Data Collection}
\label{subsec:pre_implementation_data}
% ------------------------------------------------------------

To design, train, and validate TrapPot's AI module before live deployment, secondary data from reputable public and academic sources is used \cite{hindy2023comprehensive,dekeersmaeker2023survey,alex2023comprehensive}. This approach is aligned with established research methodologies in IoT security and ensures the system is grounded in real-world attack patterns \cite{rodriguezgomez2025systematic,lanz2025optimizing}.

\textbf{Sources of Secondary Data:}

\textbf{Academic Publications \& Research Datasets}

\begin{itemize}
    \item Lee et al. (2021) – Provided labeled attack logs from Cowrie honeypots in IoT smart factory environments, classifying brute-force, scanning, and malware download attacks with 96\% accuracy using Random Forest \cite{lee2021classification}.
    \item Saputro et al. (2021) – Supplied performance benchmarks and attack traces from medium-interaction IoT honeypots, confirming Cowrie's low resource overhead (<6.3\% CPU) \cite{9359711}.
    \item Saied et al. (2023) – Offered comparative datasets for IoT botnet classification, demonstrating Random Forest's superiority (99.99\% accuracy) for multi-class attack detection \cite{saied2023comparative}.
    \item Alajrami \& Abu-Naser (2025) – Shared SSH/Telnet attack datasets integrated with honeypot logs, used for testing multi-algorithm IDS performance \cite{alajrami2025aipowered}.
\end{itemize}

\textbf{Public IoT Security Datasets}

\begin{itemize}
    \item Mirai Botnet Traces – Historical attack data from Mirai and variants (e.g., Gafgyt, Mozi), focusing on Telnet/SSH credential-based attacks \cite{203628,herwig2021analyzing}.
    \item Cowrie Public Logs – Open-source honeypot logs capturing SSH/Telnet sessions, used for feature extraction and model pre-training \cite{9071014,9558948}.
    \item IoT-23 \& CIC-IoT-2023 Datasets – Labeled network traffic datasets containing IoT-specific attacks, useful for validating feature relevance and model generalization \cite{dekeersmaeker2023survey,alex2023comprehensive,christy2024iot}.
\end{itemize}

\textbf{Technical Documentation \& Cybersecurity Reports}

\begin{itemize}
    \item Cowrie Documentation – Detailed JSON log structures, command emulation lists, and session metadata formats \cite{9071014,log2evt2025}.
    \item ELK Stack Guides – Logstash pipelines and Elasticsearch mappings for parsing and storing honeypot data \cite{9001068}.
    \item Threat Intelligence Reports – Analyses of IoT attack trends (e.g., scanning patterns, malware delivery methods) from CrowdStrike, Palo Alto, and academic surveys \cite{mahmood2024exploring,islam2023large}.
\end{itemize}

\section{Data Analysis}
\label{sec:data_analysis}
% ============================================================

% ------------------------------------------------------------
\subsection{Data Acquisition and Preprocessing Pipeline}
\label{subsec:preprocessing_pipeline}
% ------------------------------------------------------------

Data the data acquisition pipeline relies on Zeek transport-layer outputs. Cowrie acts as the high-interaction deception trap, while Zeek monitors network connections, generating the tabular data stream fed into the AI. \cite{9071014,log2evt2025}. The ELK Stack (Elasticsearch, Logstash, Kibana) functions as the main data processing system which handles both data processing and storage operations \cite{9001068}. The configuration of Logstash enables it to process raw JSON records which results in proper extraction of essential fields that include session identifiers and timestamps and src\_ip addresses and input data which includes command and credential attempt information \cite{9001068,log2evt2025}.

\textbf{Data Cleaning and Organization:} To maintain data integrity, records are indexed by their unique session ID and event timestamp to identify unique records which prevents duplicate entries while keeping the data accurate as described in \cite{rodriguezgomez2025systematic}. The system uses categorical imputation to handle missing data which appears in command or input fields because users experience sudden session interruptions during feature calculation \cite{ahsan2024effects,mary2024network}.

\textbf{Ground Truth Labeling Strategy:} As a supervised learning project, accurate labeling is paramount \cite{hopf2021supervised,dimauro2021supervised}. A hierarchical strategy is applied to assign one of the five target attack types (Brute Force, Scanning, Malware Download, Command Execution, Normal attack) to each session \cite{lee2021classification,karim2024efficient}:

\begin{itemize}
    \item Malware Download is identified by the presence of data in the urls field \cite{9071014}.
    \item Command Execution requires evidence of a successful login (is\_success: true) followed by the execution of subsequent commands \cite{9071014,9558948}.
    \item Brute Force is labeled when there is a high frequency of failed login attempts (is\_success: false) \cite{203628,herwig2021analyzing}.
    \item Scanning is categorized by low-interaction sessions exhibiting a pattern of port or service probing across a broad IP range \cite{zhang2023identification,suareztangil2023stories}.
    \item Normal attack Defined as benign baseline operational events. These interactions represent expected network overhead sequences, minor automated polling, or protocol checks characterized by minimal packet counts and predictable transport signatures mapping strictly to trusted internal application protocols \cite{lee2021classification}.
\end{itemize}

% ------------------------------------------------------------
\subsection{Feature Engineering and Selection}
\label{subsec:feature_engineering}
% ------------------------------------------------------------

Feature engineering transforms the categorical and sequential log data into a small, powerful set of numerical features that effectively discriminate between the five attack classes \cite{dimauro2021supervised,christy2024iot}. The creation of these features is essential for feeding meaningful data to the Random Forest model and is critical for distinguishing subtle differences between malicious activities \cite{hopf2021supervised,ahsan2024effects}. The feature set is designed to be interpretable, avoiding complex dimensionality reduction techniques to maintain clarity in the prototype \cite{lee2021classification,dimauro2021supervised}.

The primary features engineered from the raw session logs include metrics related to authentication, session activity, command usage, and malware transfer \cite{christy2024iot,karim2024efficient}:

\textbf{Authentication Metrics} include the Unique Login Attempts Count per session. A high value for this metric is a direct indicator of automated brute-force activity trying many credential pairs \cite{203628,herwig2021analyzing,9558948}.

\textbf{Session Metrics} are recorded by the Session Duration in seconds. I look at the Session Duration in seconds. The analyst uses the Session Duration in seconds to separate automated scanning attempts, which have duration from longer interactive command execution attacks, which have extended duration \cite{lee2021classification,karim2024efficient}.

\textbf{Command Metrics} focus on the attacker's post-login behavior, specifically the Executable Command Ratio. This is the percentage of executed commands that are recognized as valid operating system commands versus invalid or empty inputs, providing strong evidence of successful Command Execution \cite{9071014,dimauro2021supervised}.

\textbf{Malware Metrics} are distilled into the File Transfer Flag (Binary), a simple Boolean indicator that flags the presence of any file download or upload activity, serving as the primary numerical identifier for the Malware Download attack class \cite{9071014,9558948}.

\textbf{Activity Metrics} such, as Input Entropy measure the randomness and the diversity of input strings. Activity Metrics and Input Entropy help the detection process human interaction, from the rigid pre‑programmed inputs that automated botnet scripts use \cite{dimauro2021supervised,christy2024iot}.

The feature set is high value. The team will use the feature set directly for model training. The feature set fits the project goal of operation and high performance \cite{lee2021classification,saied2023comparative,karim2024efficient}.

% ------------------------------------------------------------
\subsection{Machine Learning Model Training and Optimization}
\label{subsec:model_training}
% ------------------------------------------------------------

The analysis relies exclusively on the Random Forest Classifier, a choice validated by the literature for its proven high accuracy (96–99\%) and low inference latency in IoT intrusion detection \cite{lee2021classification,saied2023comparative,karim2024efficient,christy2024iot}. The curated dataset will be partitioned into an 80\% training set and a 20\% hold-out testing set \cite{optimized2024arxiv,karim2024efficient}. Model robustness during training will be confirmed using 5-Fold Cross-Validation to iteratively evaluate and refine performance metrics across the entire training data \cite{pan2024xgboost,optimized2024arxiv}. To mitigate the severe dataset imbalance (where benign traffic typically outweighs attack samples), the primary mitigation strategy will be the use of Class Weighting within the Random Forest model \cite{9213728,9912910,abdulsalam2023multiclass}. This technique assigns a higher penalty to misclassification errors involving the less frequent attack classes (e.g., Malware Download), ensuring the model is not biased toward the majority classes \cite{9213728,abdulsalam2023multiclass,wardana2024federated}. Hyperparameters, specifically the number of decision trees (n\_estimators) and maximum tree depth (max\_depth), will be optimized using a Grid Search to achieve the project's minimum 99\% classification accuracy goal \cite{pan2024xgboost,optimized2024arxiv}.

\section{Validation Plan}
\label{sec:validation_plan}
% ============================================================

The validation plan for the TrapPot system is designed to objectively measure both the analytical accuracy of the machine learning classifier and the operational feasibility of the system as a real-time threat intelligence solution \cite{confusion2023pmc,mary2024network}. The overall measure of success is achieving the project objective of a classification accuracy exceeding 99\% \cite{lee2021classification,karim2024efficient,kalaria2024iotpredictor}.

% ------------------------------------------------------------
\subsection{Classification Performance Metrics}
\label{subsec:performance_metrics}
% ------------------------------------------------------------

While overall accuracy serves as a primary goal indicator, the security applications require additional evaluation methods because dataset imbalance affects security systems according to \cite{confusion2023pmc,mary2024network}.

\textbf{Primary Metric:} The main performance metric for this project depends on Accuracy to check if the 99\% goal is achieved through the evaluation of correct prediction outcomes for all five attack categories \cite{karim2024efficient,mary2024network}.

\textbf{Secondary Critical Metric:} The F1-Score will be tracked and reported alongside accuracy \cite{confusion2023pmc,karim2024efficient}. The F1-Score uses the harmonic mean of Precision and Recall to achieve a balanced assessment which minimizes the effect of models that struggle to identify minority classes \cite{confusion2023pmc,9213728}.  For TrapPot, a high F1-Score is essential to ensure that less frequent but high-impact attacks (like Malware Download or Command Execution) are not overlooked in favor of the more numerous Brute Force attempts \cite{lee2021classification,9213728}.

% ------------------------------------------------------------
\subsection{Validation Dataset and Testing Method}
\label{subsec:validation_dataset}
% ------------------------------------------------------------

The final validation test will be performed after the model training and tuning process is complete \cite{optimized2024arxiv,confusion2023pmc}.

\textbf{Validation Data:} The system will use the 20\% hold-out dataset that was strictly separated during the data preparation phase \cite{optimized2024arxiv,karim2024efficient}. This data represents real-world attack attempts captured by the Cowrie honeypot over the specified collection period but has never been seen by the Random Forest model during training, ensuring an unbiased measure of generalization \cite{lee2021classification,saied2023comparative,kalaria2024iotpredictor}.

\textbf{Data Collection Duration:} The model's training and final testing data will be derived from logs collected over a 30-day period of continuous operation of the Cowrie honeypot in the isolated network environment \cite{lee2021classification,suareztangil2023stories}. This duration provides a robust and diverse sample of global attack traffic targeting Telnet and SSH \cite{herwig2021analyzing,9558948}.

\textbf{Methodology:} The trained Random Forest model will be applied to the 20\% hold-out set, and the resulting predictions will be compared against the ground truth labels established in the preprocessing pipeline (Section 3.3.1) \cite{confusion2023pmc,mary2024network}.

% ------------------------------------------------------------
\subsection{Operational Success Criteria}
\label{subsec:operational_criteria}
% ------------------------------------------------------------

Beyond analytical metrics, The system needs to show its ability to operate as a real-time threat intelligence system through operational readiness according to \cite{rodriguezgomez2025systematic,lanz2025optimizing}. Two key system-level criteria must be met:

\textbf{Real-Time Inference Latency:} The system must demonstrate low latency in classifying new attack data as it arrives \cite{lee2021classification,karim2024efficient}. The time taken for the trained Random Forest model to process a new session's features and output a classification prediction must be minimal, supporting the project's goal of real-time monitoring via the ELK Stack \cite{9001068,karim2024efficient}.

\textbf{Resource Efficiency:} The overall system, including the Cowrie honeypot, the Logstash processing, and the Random Forest classification module, must operate with low computational overhead (CPU and memory usage) \cite{9359711,alzahrani2025lightweight}. This validates the system's suitability for feasible deployment within educational or non-enterprise IT environments \cite{franco2021survey,morozov2024sweet}.

\section{Research Ethics}
\label{sec:research_ethics}
% ============================================================

The TrapPot project operates under an ethical framework which bases its operations on Do No Harm principles while focusing on two main objectives to protect network security and maintain data integrity and handle adversary data properly \cite{commey2025blockchain,franco2021survey}. The project requires passive monitoring of actual cyber threats in real networks since it does not involve testing with human participants thus network separation and data protection become the main ethical concerns \cite{franco2021survey,alawadi2022_iot_honeypots}.

% ------------------------------------------------------------
\subsection{Network Isolation and Containment}
\label{subsec:network_isolation}
% ------------------------------------------------------------

Network safety is the paramount ethical and technical consideration \cite{commey2025blockchain,albaseer2024fedpot}. The Cowrie honeypot, while designed to capture attack data, must never pose a risk to the university's production network or serve as a launchpad for attacks against third parties \cite{commey2025blockchain,fan2019honeydoc}.

\textbf{Strict Segregation:} The TrapPot system is deployed within a dedicated, highly isolated network segment or Virtual Local Area Network (VLAN), ensuring complete separation from all internal production infrastructure \cite{albaseer2024fedpot,fan2019honeydoc}.

\textbf{Preventing Egress:} The main containment method depends on firewall egress rules which operate with absolute strictness according to Commey and his team \cite{commey2025blockchain}. The rules prevent all outgoing traffic which starts from the honeypot host because they block adversaries from using the compromised simulated environment to attack external targets or move to different targets.

\textbf{Sandbox Security:} The medium-interaction nature of the Cowrie platform provides an ethical sandbox \cite{9071014,9359711}. The attacker's commands run inside a secured environment which prevents them from executing any host operating system code thus preventing any possible exploitation attempts.

% ------------------------------------------------------------
\subsection{Data Handling and Anonymization}
\label{subsec:data_handling}
% ------------------------------------------------------------

The ethical handling of captured data, which includes potentially identifiable information, is mandatory \cite{rodriguezgomez2025systematic}.

\textbf{Data Minimization:} The project maintains only the essential data needed for threat intelligence and classification which includes Source IP addresses (src\_ip) and timestamps and credentials tried and executed commands \cite{rodriguezgomez2025systematic,lee2021classification}. The research team can analyze human activities through data processing of their gathered information to achieve their project goals \cite{9071014}.

\textbf{Storage and Access:} All log data stored within the ELK Stack and research servers is securely restricted, accessible exclusively to the project team for research purposes \cite{9001068}.

\textbf{Anonymity in Reporting:} Any public presentation or reporting of the results, including The final project documentation together with all public presentations of results will present statistics and geographic data through aggregated and anonymized forms \cite{rodriguezgomez2025systematic}. The survey by Franco 2021 indicates that researchers will never disclose individual source IP addresses as raw identifiable data to public domains \cite{franco2021survey}.

% ------------------------------------------------------------
\subsection{Regulatory Compliance and Log Management}
\label{subsec:regulatory_compliance}
% ------------------------------------------------------------

\textbf{Institutional Approval:} The institutional IT Department needs to approve the 30-day data collection period to confirm that the deployment and network isolation plan complies with all security and network policies of the university \cite{franco2021survey,alawadi2022_iot_honeypots}.

\textbf{Data Disposal:} Upon the successful completion and submission of the project, all raw, identifiable logs captured by the Cowrie honeypot will be securely purged from the ELK Stack and associated storage infrastructure \cite{rodriguezgomez2025systematic}. Only final, anonymized, and aggregated statistical summaries will be retained for future comparative research \cite{franco2021survey,rodriguezgomez2025systematic}.
