% ============================================================
\section{Research Design}
\label{sec:research_design}
% ============================================================

TrapPot is not a survey. It is a built system, and that changes what kind of research design makes sense \cite{franco2021survey,rodriguezgomez2025systematic}.

% ------------------------------------------------------------
\subsection{Case Study Design}
\label{subsec:case_study}
% ------------------------------------------------------------

We used a case study design centered on one deployment: TrapPot running in an isolated local lab environment \cite{albaseer2024fedpot,commey2025blockchain}. That scope is intentional. A case study lets us examine how the system actually behaves rather than making shallow comparisons across many implementations we do not fully control.

What a case study lets us observe \cite{rodriguezgomez2025systematic,lanz2025optimizing}:

\begin{itemize}
\item how attackers (or simulated attacker traffic) interact with emulated SSH and Telnet services \cite{9071014,9558948,famera2025mirai},
\item how logs flow through the full ELK pipeline, from Cowrie JSON to Elasticsearch index \cite{9001068,log2evt2025}, and
\item how the Random Forest classifier performs on all five classes: brute-force, scanning, malware download, command execution, and normal traffic \cite{lee2021classification,karim2024efficient,saied2023comparative}.
\end{itemize}

A broader comparative study across multiple honeypot deployments would have been harder to control and would have produced results harder to interpret \cite{franco2021survey,rodriguezgomez2025systematic}. We therefore chose one deployment with complete observation.

% ------------------------------------------------------------
\subsection{Quantitative Approach}
\label{subsec:quantitative}
% ------------------------------------------------------------

Everything TrapPot produces is a number or a structured record \cite{9071014,log2evt2025}. Cowrie logs authentication attempts and command counts. Zeek logs ports, byte counts, packet counts, duration, protocol, service, and connection state. The classifier produces class labels and accuracy scores. All of it can be measured \cite{dimauro2021supervised,ahsan2024effects}.

The research follows four steps \cite{lee2021classification,rodriguezgomez2025systematic}:

\begin{enumerate}
\item Deploy TrapPot with Cowrie, Zeek, the Random Forest detector, and the ELK Stack in an isolated network \cite{9359711,commey2025blockchain}.
\item Collect data: IoT-23 for training, live Cowrie/Zeek logs for operational testing \cite{lee2021classification,suareztangil2023stories}.
\item Train and test the Random Forest model, evaluate against the 99\% accuracy target \cite{saied2023comparative,karim2024efficient}.
\item Present results through tables, confusion matrix, and Kibana dashboard \cite{confusion2023pmc,mary2024network}.
\end{enumerate}

\section{Data Collection}
\label{sec:data_collection}
% ============================================================

Two data sources. Two purposes. They never mix \cite{rodriguezgomez2025systematic,lanz2025optimizing}.

% ------------------------------------------------------------
\subsection{Pre-Implementation Data}
\label{subsec:pre_implementation_data}
% ------------------------------------------------------------

Before we deployed anything, we needed a dataset to train the model on. We chose IoT-23 \cite{iot23dataset}, a labeled collection of IoT network traffic flows structured in Zeek-compatible format and used in intrusion detection research \cite{dekeersmaeker2023survey,alex2023comprehensive}. The Zeek compatibility is what made it the right choice here: our live detector reads \texttt{conn.log} features, and IoT-23 maps directly to that format.

\textbf{Academic papers and datasets we relied on:}

\begin{itemize}
\item Lee et al.\ (2021): Cowrie-based IoT honeypot with Random Forest, hitting 96\% accuracy on brute-force, scanning, and malware download in a smart factory environment. This was the closest prior work to TrapPot's setup \cite{lee2021classification}.
\item Saputro et al.\ (2021): performance benchmarks for Cowrie deployments, confirming CPU overhead below 6.3\% in containerized environments \cite{9359711}.
\item Saied et al.\ (2023): comparative study of Random Forest for IoT botnet classification, reporting 99.99\% accuracy on multi-class detection \cite{saied2023comparative}.
\item Alajrami \& Abu-Naser (2025): SSH/Telnet honeypot IDS combining Cowrie with several classifiers \cite{alajrami2025aipowered}.
\end{itemize}

\textbf{Public datasets:}

\begin{itemize}
\item IoT-23 and CIC-IoT-2023 for labeled traffic flows \cite{iot23dataset,dekeersmaeker2023survey,christy2024iot}.
\item Mirai botnet traces for credential-stuffing and Telnet/SSH attack patterns \cite{203628,famera2025mirai}.
\item Cowrie public session logs for feature extraction reference \cite{9071014,9558948}.
\end{itemize}

\textbf{Documentation:}

\begin{itemize}
\item Cowrie's JSON log format (what fields are available, what they mean) \cite{9071014,log2evt2025}.
\item ELK Stack configuration guides for Logstash pipeline and Elasticsearch mappings \cite{9001068}.
\item IoT threat reports for context on what attack patterns we should expect to see \cite{mahmood2024exploring,islam2023large}.
\end{itemize}

% ------------------------------------------------------------
\subsection{Post-Implementation Data}
\label{subsec:post_implementation_data}
% ------------------------------------------------------------

Once the system was running, it collected its own data in real time \cite{lee2021classification,suareztangil2023stories}. Cowrie logs each session. Zeek writes \texttt{conn.log} and \texttt{ssh.log}. The AI detector outputs to \texttt{detections.json}. Logstash picks all of it up and sends it to Elasticsearch.

This live data has three uses: feeding the classifier in real time, supporting future model retraining with actual honeypot sessions, and providing the operational evidence discussed in Chapter~\ref{sec:analysis_result}. The implementation detail of this phase is in Chapter~\ref{sec:integration_deployment}.

\section{Data Analysis}
\label{sec:data_analysis}
% ============================================================

% ------------------------------------------------------------
\subsection{Preprocessing Pipeline}
\label{subsec:preprocessing_pipeline}
% ------------------------------------------------------------

The pipeline is Zeek-first. Cowrie handles deception; Zeek handles conversion. Every SSH or Telnet connection that hits Cowrie produces a row in \texttt{conn.log}, and that row is what the AI detector actually reads \cite{9071014,log2evt2025}.

Logstash extracts the fields we need: session ID, timestamp, source IP, commands, credential attempts \cite{9001068,log2evt2025}. Records get organized by session ID and event timestamp so we can track the relationship between what an attacker typed and what appeared on the wire. Missing fields (empty command inputs, incomplete sessions) get handled during feature calculation. We do not drop those records; we fill the missing values with defaults so the model always receives a complete feature vector \cite{ahsan2024effects,mary2024network}.

\textbf{Labeling strategy.} Five classes. The hierarchy runs from most specific to least specific \cite{lee2021classification,karim2024efficient}:

\begin{itemize}
\item \textbf{Malware Download}: URLs field is non-empty; the session involved a file transfer attempt \cite{9071014}.
\item \textbf{Command Execution}: successful login followed by at least one shell command \cite{9071014,9558948}.
\item \textbf{Brute Force}: high count of failed login attempts within a short window \cite{203628,famera2025mirai}.
\item \textbf{Scanning}: low-interaction sessions, port or service probing, no successful auth \cite{zhang2023identification,suareztangil2023stories}.
\item \textbf{Normal}: baseline traffic, minimal packet counts, predictable transport signatures, and no attack indicators \cite{lee2021classification}.
\end{itemize}

% ------------------------------------------------------------
\subsection{Feature Engineering}
\label{subsec:feature_engineering}
% ------------------------------------------------------------

The live detector uses twelve features. We kept the feature set compact because additional features create more ways for the pipeline to break when Zeek's output format changes, and we wanted predictions that we could reason about when a result looked wrong \cite{dimauro2021supervised,christy2024iot}.

The features group into five categories \cite{christy2024iot,karim2024efficient}:

\textbf{Authentication}: number of unique login attempts per session. High counts point to automated credential stuffing \cite{203628,famera2025mirai,9558948}.

\textbf{Session behavior}: duration in seconds. This feature is useful because a scanner that never authenticates stays for a fraction of a second, while a command-execution session can run for minutes \cite{lee2021classification,karim2024efficient}.

\textbf{Command usage}: ratio of recognized OS commands to total input. After login, attackers run real commands; bots that never authenticate produce invalid input or nothing \cite{9071014,dimauro2021supervised}.

\textbf{Malware indicator}: binary flag, 1 if there was a file transfer attempt. This feature separates the Malware Download class from the other classes \cite{9071014,9558948}.

\textbf{Network}: source and destination ports, protocol, service, duration, byte counts, packet counts, connection state. All of these come from Zeek's \texttt{conn.log}. These are the features that let the model classify traffic from network observations without needing to read a Cowrie session log.

We avoided dimensionality reduction. The team needed to understand what each feature meant, and 12 features is small enough that every feature can be explained in one sentence. That matters for a thesis defense.

% ------------------------------------------------------------
\subsection{Model Training and Optimization}
\label{subsec:model_training}
% ------------------------------------------------------------

We used Random Forest because Lee et al.\ reported 96\% accuracy on Cowrie-based IoT honeypot data, and Saied et al.\ reported 99.99\% on IoT botnet classification \cite{lee2021classification,saied2023comparative}. A well-understood algorithm with a strong track record on similar data was a better fit than a more complex algorithm that would be harder to explain \cite{karim2024efficient,christy2024iot}.

The setup: 80\% training, 20\% hold-out, five-fold cross-validation during training \cite{optimized2024arxiv,karim2024efficient}.

The main training problem was class imbalance. IoT-23 has far more brute-force records than malware download records. The first three training runs scored around 72\%, and the model was effectively ignoring the smaller classes \cite{9213728,9912910,abdulsalam2023multiclass}. We applied SMOTE oversampling to balance the training distribution, combined with class weighting. After that, the hold-out accuracy was 99.59\%.

Hyperparameters, including number of trees and maximum depth, were tuned to hit the 99\% accuracy target we set at the start \cite{pan2024xgboost,optimized2024arxiv}. The final model was saved as \texttt{trappot\_model.pkl} with \texttt{label\_encoder.pkl} for the live detector container.

\section{Validation Plan}
\label{sec:validation_plan}
% ============================================================

Two things to validate: the model's classification performance, and the system's operational behavior \cite{confusion2023pmc,mary2024network}. Both matter. A 99.59\% model that cannot connect to a live Zeek log is not a finished system.

% ------------------------------------------------------------
\subsection{Classification Metrics}
\label{subsec:performance_metrics}
% ------------------------------------------------------------

Primary metric: accuracy on the 20\% hold-out. Target: 99\% \cite{karim2024efficient,mary2024network,lee2021classification}.

But accuracy alone is not enough for a security classifier. If the model predicts brute-force for 98\% of traffic and happens to be right 98\% of the time, the remaining classes are invisible. So we also report F1-score per class, which balances precision and recall and makes it harder to hide behind a dominant class \cite{confusion2023pmc,karim2024efficient}. The confusion matrix shows where the model confuses one class for another, which is more useful than a single accuracy number \cite{confusion2023pmc,mary2024network,lee2021classification,9213728}.

% ------------------------------------------------------------
\subsection{Validation Dataset and Method}
\label{subsec:validation_dataset}
% ------------------------------------------------------------

The hold-out set is 20\% of the prepared IoT-23 data, separated before training begins and never touched until final evaluation \cite{optimized2024arxiv,karim2024efficient}. The model applies its predictions to those records and we compare against the ground truth labels from Section~\ref{subsec:preprocessing_pipeline}.

For the operational side, we ran a manual verification session against the live deployment: SSH connection to port 2222, five commands (\texttt{whoami}, \texttt{pwd}, \texttt{ls -la}, \texttt{cat /etc/passwd}, \texttt{wget}), then checked every downstream component for the expected output. The full 30-day data collection window from the original design study remains available for future work with a properly isolated network \cite{lee2021classification,suareztangil2023stories,famera2025mirai,9558948}.

% ------------------------------------------------------------
\subsection{Operational Criteria}
\label{subsec:operational_criteria}
% ------------------------------------------------------------

Two system-level requirements \cite{rodriguezgomez2025systematic,lanz2025optimizing}:

\textbf{Inference latency.} The detector must classify new \texttt{conn.log} entries fast enough that Kibana can show near-real-time results. Random Forest inference is fast; Lee et al.\ reported 0.1~ms per record \cite{lee2021classification}. We still verified the live detector path during testing.

\textbf{Resource use.} The full stack, Cowrie, Zeek, AI detector, Elasticsearch, Logstash, and Kibana, must run on a single lab machine without crashing or throttling \cite{9359711,alzahrani2025lightweight}. Cowrie and the detector are lightweight. The ELK containers (Elastic Stack 9.4.0) require more memory, and the host machine needs to provide it \cite{franco2021survey,morozov2024sweet}.

\section{Research Ethics}
\label{sec:research_ethics}
% ============================================================

A honeypot is, by design, something that attracts malicious activity. That creates an ethical obligation: the research must not put other systems at risk to collect its data \cite{commey2025blockchain,franco2021survey}.

% ------------------------------------------------------------
\subsection{Network Isolation}
\label{subsec:network_isolation}
% ------------------------------------------------------------

TrapPot runs inside Docker's bridge network with no public exposure \cite{albaseer2024fedpot,fan2019honeydoc}. Ports 2222, 2223, and 5601 are the only exposed surfaces. Cowrie's medium-interaction design means attacker commands run inside the honeypot and do not reach the host OS \cite{9071014,9359711}. Firewall rules on the host prevent outgoing traffic from the honeypot containers so that even if an attacker gained unexpected capabilities, they could not pivot to external targets \cite{commey2025blockchain}.

This system should only ever be connected to an approved, isolated lab network. Not the university's production network. Not a home router. Somewhere with a firewall between TrapPot and the rest of the world.

% ------------------------------------------------------------
\subsection{Data Handling}
\label{subsec:data_handling}
% ------------------------------------------------------------

Cowrie logs include source IP addresses and credential attempts, which may be real data from real or simulated attackers \cite{rodriguezgomez2025systematic,lee2021classification}. We store only what is needed for classification and analysis. Raw IP addresses are not published in the thesis or any public report \cite{franco2021survey,rodriguezgomez2025systematic}.

Access to the ELK indices is limited to the project team. After the project is submitted, raw identifiable logs should be purged and only anonymized aggregates retained \cite{9001068,rodriguezgomez2025systematic}.

% ------------------------------------------------------------
\subsection{Institutional Compliance}
\label{subsec:regulatory_compliance}
% ------------------------------------------------------------

Any live data collection beyond the current lab test requires approval from the relevant university or lab authority \cite{franco2021survey,alawadi2022_iot_honeypots}. The project design keeps all network activity within a controlled environment, and no human subjects are involved. The main compliance requirement is the network isolation plan described above.
