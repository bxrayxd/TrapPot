This chapter summarizes the main results of the TrapPot project and reflects on the results achieved during the implementation and testing phases. It also describes limitations of the current version of the system and presents future improvements that can make TrapPot a stronger cybersecurity research and learning platform.

\section{Discussion of Results} 
\label{sec:discussion_of_results}

The result of the project shows that TrapPot has been successfully deployed as a Docker based honeypot and network monitoring system. The system managed to integrate several components into one working environment. These components were Cowrie for SSH and Telnet honeypot interaction, Zeek for network traffic monitoring, a Random Forest-based AI detector for classification and the ELK Stack for log storage and visualization.

In the test, Cowrie was able to accept ssh connections on port 2222 and serve the user with a fake Linux environment. The honeypot logged login attempts, the execution of commands and attempts to download files. This proves that the honeypot part fulfilled its main purpose of attracting and capturing attacker-like behavior without actually providing a real production system.

Zeek also played an important role in the system. It monitored the traffic generated by the Cowrie session and generated well-structured logs such as \texttt{conn.log} and \texttt{ssh.log}. The logs provided useful network-level information, such as ports, protocols, services, duration, byte counts, packet counts, and connection states. This made Zeek a central point of connection between the honeypot and the AI detector.

The AI detector could read the \texttt{conn.log} file of Zeek and extract the required twelve network traffic features and apply the trained Random Forest model. The detector generated live classification alerts and wrote detection records to \texttt{detections.json}. This indicates that the machine learning component was not trained in isolation, but was instead integrated into the live TrapPot pipeline.

The collected data were analyzed and presented using the ELK Stack. The logs were processed by Logstash and shipped to Elasticsearch and Kibana provided data views and the TrapPot dashboard. This allowed a more convenient way of checking system activity through a browser interface instead of just relying on terminal logs.

The overall project results show that TrapPot has achieved its primary goal as a proof-of-concept system. It shows how honeypot technology, network monitoring, machine learning, and visualization tools can be combined to support IoT attack monitoring and analysis.


\subsection{Contributions} % (fold)
\label{sub:contributions}

The main innovation of TrapPot is a honeypot-based integrated monitoring system, which integrates the capture of attacker interactions, network traffic analysis, machine learning classification and log visualization. The system brings together Cowrie, Zeek, the AI detector and ELK Stack in a single Docker-based environment, so that the activity collected can be analyzed from both honeypot and network perspectives.

TrapPot also provides a handy AI detection workflow for the project environment. The Random Forest model was implemented in the live detector which reads Zeek's \texttt{conn.log}, extracts the twelve network features required and produces classification alerts during runtime. This shows that the machine learning model was trained on its own and linked to the deployed system pipeline.

Another contribution is the approach of the portable deployment. It is easier to start the system and reproduce it on other machines using Docker Compose without having to install each component manually. Thus, the project is well suited for demonstration, testing and future development in a controlled lab environment.

\section{Limitations} % (fold)
\label{sec:limitations}

While TrapPot showed promising results, there were several limitations encountered during its implementation and testing:

\begin{itemize}
   \item \textbf{Controlled Testing Environment:} The system was tested in a local lab environment not exposed to public attacker traffic. This was done for safety and ethical reasons, but it means that the system was not tested on a large amount of real-world attack activity. 

   \item \textbf{Dataset-Based Model Accuracy:} The Random Forest model got accuracy as approximately 99.59\% for the prepared dataset IoT-23. However, high accuracy on a dataset does not guarantee the same performance in a live deployment, especially when the real traffic patterns deviate from the training data.

   \item \textbf{Limited Live Attack Variety:} Live testing primarily consisted of manual interaction using SSH and running simple commands. Further attack scenarios are needed to evaluate the performance of the detector with scanning, brute-force, malware downloading behavior, and command execution patterns.
   
   \item \textbf{Feature Scope:} The current AI detector is mainly based on Zeek \texttt{conn.log} features. These features help in network-based classification but they do not fully represent detailed Cowrie session behavior such as command sequences, login patterns, and downloaded file indicators.

   \item \textbf{Resource Requirements:} Elasticsearch, Logstash, and Kibana have powerful storage and visualization capabilities, but they also require more memory and processing power than the honeypot and AI detector components. This can be detrimental in a long term deployment or with a high traffic volume.
\end{itemize}

Nevertheless, the project is still useful, but it shows where the system can be improved in the next versions. The current implementation works well as a working prototype and provides a good starting point for further development.

\section{Future Work} % (fold)
\label{sec:future_work}

To improve the system and extend its practical value, several areas of future work are proposed.

\subsection{Testing with Realistic Attack Traffic} 
\label{subsec:realistic_attack_traffic}

The future versions of TrapPot should be tested in a validated isolated network environment with more realistic traffic. It would help the system to collect genuine attacker behaviors in a safe and controlled manner. A larger live dataset would help to assess how well the AI detector performs under different attack conditions.

The tests could involve different types of SSH and Telnet activity, including repeated login attempts, scanning activity, automated bots, attempts to download files, and command execution. The current version has been only lightly tested by hand, and a more thorough evaluation of the system is needed.

\subsection{Combining Zeek and Cowrie Features} 
\label{subsec:combined_features}

The existing AI detector relies on twelve network-level features derived from Zeek’s \texttt{conn.log}. Future work can enhance the detection process by combining these features with Cowrie session level features. These may include the number of login attempts, commands rate, command sequences patterns, session length, signs of files being downloaded, and suspicious commands.

The Zeek and Cowrie features combined would provide the model the ability to analyze both network behavior and attacker interaction behavior. This may enhance the quality of classification and provide a more comprehensive view of malicious activity.

\subsection{Indexing AI Results in Elasticsearch} 
\label{subsec:indexing_ai_results}

The current implementation has the detector print live alerts and save detection records in \texttt{detections.json}. A good future improvement would be to send these detection results directly in Elasticsearch. This would allow Kibana to show AI predictions, detected categories and detection timestamps on the dashboard itself.

This would make the visualization layer better, because the user would be able to see the logs of Cowrie, Zeek and AI detector results in one central interface. This would also improve the usefulness of the system for reporting and long-term monitoring.

\subsection{Exploring Additional Machine Learning Models} 
\label{subsec:additional_ml_models}

The Random Forest model achieved good results on the prepared IoT-23 dataset, but it can be compared with other machine learning models in future work. Among possible models are XGBoost, Support Vector Machines, and neural network-based approaches.

Testing different models would help to identify if a different approach can improve the classification performance, reduce false predictions or deal with more complex attack patterns. Comparing models would also strengthen the evaluation of machine learning.

\subsection{Improving Deployment Security} 
\label{subsec:deployment_security}

TrapPot’s future iterations can also improve the security of deployments. That could mean stricter container isolation, more network controls in Docker, controlled access to Kibana, automated backups, and clearer log retention policies.

These improvements would make the system more suitable for a longer period of time in a controlled cybersecurity lab environment. They would also lower the risks associated with running honeypot services, especially if the system is later tested with external traffic.

\section{Conclusion} % (fold)
\label{sec:conclusion}

TrapPot shows the power of the combination of deception, monitoring and intelligent analysis in one controlled environment of cyber security. The project was created to address the challenge of understanding SSH and Telnet-based attack behavior against IoT-like systems without risking a real production device. The system achieved its main goal by providing a working proof of concept for capturing suspicious interaction, monitoring the associated network activity, analyzing traffic behavior and presenting the collected information using a dashboard. This indicates that honeypot-based environments can be used for collecting logs, but also for learning, research and early experiments with AI-assisted intrusion detection.
Overall TrapPot provides a strong basis for education in the field of cybersecurity, monitoring of attacks on IoT and for the development of future research.
