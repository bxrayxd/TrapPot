This chapter summarizes the main results of the TrapPot project and reflects on the implementation and testing outcomes. It also presents the current limitations and future improvements that can make TrapPot stronger as a cybersecurity research and learning platform.

\section{Discussion of Results}
\label{sec:discussion_of_results}

The project results show that TrapPot was successfully deployed as a Docker-based honeypot, monitoring, classification, and visualization system. The implemented environment brings together Cowrie for SSH and Telnet honeypot interaction, Zeek for network traffic monitoring, a Random Forest-based AI detector for classification, and the ELK Stack for log storage and visualization.

During testing, Cowrie accepted SSH connections on port 2222 and provided a fake Linux shell environment. The honeypot logged login attempts, executed commands, and file download attempts. This confirms that the honeypot layer fulfilled its main purpose: capturing attacker-like behavior without exposing a real production system.

Zeek also played an important role. It monitored the traffic generated by the Cowrie session and produced structured logs such as \texttt{conn.log} and \texttt{ssh.log}. These logs provided network-level information, including ports, protocols, services, duration, byte counts, packet counts, and connection states. This made Zeek the link between honeypot interaction and AI-based traffic classification.

The AI detector read Zeek's \texttt{conn.log}, extracted the required twelve network traffic features, and applied the trained Random Forest model. The detector generated live classification alerts and wrote detection records to \texttt{detections.json}. This shows that the machine learning component was integrated into the live TrapPot pipeline rather than kept as a separate offline experiment.

The collected data was processed and presented through the ELK Stack. Logstash shipped records to Elasticsearch, and Kibana provided data views and the \texttt{TrapPot Overview} dashboard. This allowed the project team to inspect system activity through a browser interface instead of relying only on terminal logs.

Overall, TrapPot achieved its primary goal as a proof-of-concept system. It shows how honeypot technology, network monitoring, machine learning, and visualization tools can be combined to support IoT attack monitoring and analysis.


\subsection{Contributions}
\label{sub:contributions}

The main contribution of TrapPot is an integrated monitoring workflow that combines attacker-interaction capture, network traffic analysis, machine learning classification, and log visualization. The system connects Cowrie, Zeek, the AI detector, and the ELK Stack inside one Docker-based environment, allowing collected activity to be analyzed from both honeypot and network perspectives.

TrapPot also provides a practical AI detection workflow for the project environment. The Random Forest model is used by the live detector, which reads Zeek's \texttt{conn.log}, extracts the twelve required network features, and produces classification alerts at runtime. This demonstrates that the trained model was connected to the deployed system pipeline.

Another contribution is portability. Docker Compose makes it easier to start the system and reproduce it on other machines without manually installing and configuring each component. This makes TrapPot suitable for demonstrations, testing, and future development in a controlled lab environment.

\section{Limitations}
\label{sec:limitations}

Although TrapPot produced promising results, several limitations remain:

\begin{itemize}
   \item \textbf{Controlled Testing Environment:} The system was tested in a local lab environment and was not exposed to public attacker traffic. This was necessary for safety and ethical reasons, but it means the system was not tested against a large volume of real-world attack activity.

   \item \textbf{Dataset-Based Model Accuracy:} The Random Forest model achieved approximately 99.59\% accuracy on the prepared IoT-23 dataset. However, high dataset accuracy does not guarantee identical performance in a live deployment, especially when real traffic differs from the training data.

   \item \textbf{Limited Live Attack Variety:} Live testing mainly used manual SSH interaction and simple commands. Additional attack scenarios are needed to evaluate scanning, brute-force attempts, malware download behavior, and command execution patterns more deeply.

   \item \textbf{Feature Scope:} The current AI detector mainly uses Zeek \texttt{conn.log} features. These features are useful for network-based classification, but they do not fully represent detailed Cowrie session behavior such as command sequences, login-attempt patterns, and downloaded file indicators.

   \item \textbf{Resource Requirements:} Elasticsearch, Logstash, and Kibana provide powerful storage and visualization capabilities, but they require more memory and processing power than the honeypot and AI detector components. Long-term deployment or high traffic volume may require performance tuning.
\end{itemize}

These limitations do not remove the value of the project. Instead, they define the next steps required to move TrapPot from a working prototype toward a stronger research and lab deployment.

\section{Future Work}
\label{sec:future_work}

Several improvements can extend the practical value of TrapPot.

\subsection{Testing with Realistic Attack Traffic}
\label{subsec:realistic_attack_traffic}

Future versions of TrapPot should be tested in an approved isolated network environment with more realistic traffic. This would allow the system to collect genuine attacker behavior safely. A larger live dataset would also help evaluate how well the AI detector performs under different attack conditions.

Future tests should include repeated login attempts, scanning activity, automated bots, file download attempts, and command execution. The current version was tested manually, so broader validation is needed.

\subsection{Combining Zeek and Cowrie Features}
\label{subsec:combined_features}

The current AI detector relies on twelve network-level features derived from Zeek's \texttt{conn.log}. Future work can improve classification by combining these features with Cowrie session-level features. Examples include number of login attempts, command rate, command sequence patterns, session length, file download indicators, and suspicious command usage.

Combining Zeek and Cowrie features would allow the model to analyze both network behavior and attacker interaction behavior. This may improve classification quality and provide a more complete view of malicious activity.

\subsection{Indexing AI Results in Elasticsearch}
\label{subsec:indexing_ai_results}

The current implementation prints live alerts and saves detection records in \texttt{detections.json}. A useful improvement is to send these results directly to Elasticsearch or continue strengthening the Logstash-based ingestion path for detector output. This would allow Kibana to show AI predictions, detected categories, and detection timestamps in the same dashboard as Cowrie and Zeek activity.

\subsection{Exploring Additional Machine Learning Models}
\label{subsec:additional_ml_models}

The Random Forest model achieved strong results on the prepared IoT-23 dataset, but future work can compare it with other models such as XGBoost, Support Vector Machines, and neural network-based approaches. Comparing models would help determine whether another method can reduce false predictions or handle more complex attack patterns.

\subsection{Improving Deployment Security}
\label{subsec:deployment_security}

Future iterations can improve deployment security through stricter container isolation, stronger network controls, controlled access to Kibana, backup policies, and clearer log retention rules. These improvements would make TrapPot more suitable for longer controlled lab deployments and reduce the risks associated with running honeypot services.

\section{Conclusion}
\label{sec:conclusion}

TrapPot demonstrates how deception, monitoring, machine learning, and visualization can work together in one controlled cybersecurity environment. The project addresses the challenge of understanding SSH and Telnet-based attacks against IoT-like systems without risking a real production device. It achieved its main goal by providing a working proof of concept for capturing suspicious interaction, monitoring related network activity, classifying traffic behavior, and presenting the collected information through a dashboard.

Overall, TrapPot provides a strong foundation for cybersecurity education, IoT attack monitoring, and future research development.
